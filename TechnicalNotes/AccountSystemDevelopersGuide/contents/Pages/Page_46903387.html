4. Using From the Application

4.1. Development Environment

4.1.1. Library

The account library bundled with the CTR-SDK provides the following primitive features.

  • Get IDs for identifying accounts.
  • Get basic information registered in a Nintendo Network account.
  • Authenticate users based on information registered in Nintendo Network accounts.

Other account-related features are provided by separate libraries and independent applications.

  • E-Commerce: Nintendo eShop (Application)
  • Miiverse Features: OLV Library

For more information, see the documentation for each feature or the library reference.

4.1.2. Server Environments

The following account server environments are available.

  • Development environment
  • Production environment
  • Lotcheck environment

Normally, the CTR connects to a development environment account server for development hardware and to a production environment account server for retail hardware. A Lotcheck environment is used for testing by Nintendo during Lotcheck and at other times.

Each environment is independent of the others. For instance, you cannot use a Nintendo network account created in the production environment on retail hardware from development hardware that connects to the development environment.

In addition, there is a chance that principal IDs and account IDs may be duplicated across environments because the ID space differs for each environment. However, Nintendo designed the software to prevent interactions between duplicate IDs. Servers that provide Nintendo services such as management at the account level have been split into separate account server environments. Applications do not need to keep track of the environment they connect to as long as you use a server provided by Nintendo.

Note:

There are currently no naming conventions for account IDs in the development environment. If a generated ID conflicts with other developers, get and use a different account IDt.

Note:

To prevent leaks of development information, email messages are only sent from the account server to email addresses registered on that server in the production environment. Note that when you register an account, a message saying that a confirmation email has been sent is displayed on the screen. If the email address verification process is not completed, a message prompting the user to confirm the email address may be displayed when the corresponding account is used to get a service token. To get rid of this message, log in to your account on the Account Developer Portal and edit the account settings such that the email address verification process is treated as complete.

Note:

Nintendo may delete accounts generated in the development environment that have not been used for one year or more. Please make a note of this.

4.2. Getting and Using User Information

Applications can use the account library to obtain a subset of the user information registered in system accounts and Nintendo Network accounts.

A subset of the user information in a Nintendo Network account is cached on the CTR system. You can get this information without connecting to the Nintendo Network account server each time. You can also get information for other accounts in the same system. The system updates the cache automatically.

You can get some information directly from the account server, even if it is not cached on the CTR system. However, you must connect to the network and authenticate before getting user information from the account server.

Although the application cannot update user information, users can update a subset of their information by such means as using the Nintendo Network ID Settings application in System Settings. Some of the information registered on the Nintendo Network may also be changed for operational reasons, even if the user cannot change it. If your application uses user information, you must keep track of each of these factors that could change the information. However, it is not always necessary to make strictly sure that you are using the latest user information.

4.2.1. Identifying Accounts

You can identify accounts on the CTR system by their Mii characters and several different ID values.

4.2.1.1. Account IDs

When a system account is created, the following IDs are assigned to it.

  • UUID: A 128-bit ID that conforms to the version 1 Universally Unique Identifier (UUID) specification, and is likely to be globally unique.

If you associate an account with a Nintendo Network account, the following additional IDs become available.

  • Principal IDA 32-bit globally unique ID assigned when the Nintendo Network account was created. It cannot be changed.
  • Account IDA string between 6 and 16 characters in length that a user enters and registers when creating a Nintendo Network account. The string can consist of uppercase and lowercase letters; numbers; and hyphens ("-"), periods ("."), and underscores ("_"). These IDs are globally unique, and users cannot change them at will. However, they may be changed for operational reasons. Use the principal ID if you want to permanently refer to a single user.

For more information about Nintendo Network account IDs, see the separate specifications (mentioned previously).

Note:
This principal ID is different from the principal ID managed by the Friend Presence library.
Note:

Displaying a UUID or principal ID directly in an application is prohibited. To distinguish users, display Mii faces and names, and account IDs.

Note:

The Nintendo Network system does not distinguish between uppercase and lowercase characters when comparing account IDs. Even inside your applications, ignore uppercase and lowercase differences when comparing account IDs.

4.3. Authenticating Using Nintendo Network Accounts

To use the network features provided by an application, a user must first authenticate with the Nintendo Network account server.

4.3.1. Guarantees of Authentication

Going through the authentication process on the account server guarantees the following.

4.3.1.1. COPPA Compliance

In the United States and Canada, a parental-consent procedure is required before children under 13 can create a Nintendo Network account. The account is not issued until the parental consent has been confirmed. If the user is using a Nintendo Network account, Nintendo guarantees that the user is 13 or older or is under 13 and has obtained parental consent.

Note:

The only way for a user to accept a EULA or to obtain parental consent on the CTR system is by exchanging information with a Nintendo server. If your application uses an independent server, you must implement your own EULA and COPPA support separately. For more information, see the guidelines for independent servers.

4.3.1.2. Support for Discontinued Services

If a user attempts to authenticate with a network service that has been discontinued, the server returns a response stating that the service was discontinued during the authentication phase. You can specify separately for each network service whether it is discontinued.

Nintendo performs the setting that configures a service as discontinued. Contact Nintendo support if you want to discontinue a network service.

4.3.1.3. System and Application Versions

Authentication also fails if the version of the CTR system or currently running application is not the minimum required version.

Nintendo manages and updates the required version for the system.

Contact Nintendo if you want to specify a required version for your application. Nintendo does not automatically increment the required version for each release of an application patch.

4.3.1.4. Password-Based Identification

The system validates the password entered by the user.

A password used for authentication is requested when the application accepts a Nintendo Network account service (which would primarily be the getting of a service token) and the validation period for the Nintendo Network account maintained within the CTR has expired (or when no authentication has yet occurred since the system was started). If you set the CTR to remember the password in the Nintendo Network ID Settings application or the Login Applet, authentication occurs with the remembered password.

4.3.1.5. Matching the Country of Residence

The handling of EULA acceptance and COPPA support is based on the country of residence setting in the Nintendo Network account. Authentication fails if the country of residence setting on the CTR system does not match that of the Nintendo Network account.

It is not possible to change the country of residence of the Nintendo Network account. To pass authentication, the country of residence setting on the CTR system must be configured to be the same as the Nintendo Network account.

4.3.1.6. Accepting the Latest EULA

The system verifies whether the user has accepted the latest Nintendo End-User License Agreement (EULA). If the EULA has been updated, authentication does not succeed until the user accepts the latest version.

4.3.1.7. Support for Banning

A user's access to network services may be blocked if the user repeatedly engages in illicit actions or offensive speech. (This is referred to as banning.) A user cannot authenticate a Nintendo Network account while this block is in effect.

Bans can be set by scope or period of access.

  • Scope (service)
    • All Nintendo Network services
    • Individual service only
  • Scope (user)
    • Account level
    • Console level
  • Period
    • Unlimited
    • Specified period

Only Nintendo can ban a user from all Nintendo Network services, but licensees can ban users from specific services by using the Ban Management System (BMAS).

An operator can lift a ban from a user even if the ban period has not yet expired.

4.3.2. Support Required of the Application

4.3.2.1 Getting a Service Token

The application must get service tokens to use the OLV library and to communicate with independent servers. The Login Applet is used to get service tokens. The Login Applet takes care of processes needed for account authentication like password entry and token acquisition, so there is no need for the application to handle password entry and management on its side.

For more information about token acquisition with the OLV and NEX libraries, see the manuals for those libraries.

4.3.2.2. Supporting Applications That Require the Nintendo Network ID

For applications that provide features that can only be used when a Nintendo Network ID is registered, display the following messages so that the user knows how to use the relevant features.

You have yet to register a Nintendo Network ID to this Nintendo 3DS.

To do so, use the Nintendo Network ID Settings application in the System Settings.

These messages will be added as error messages. For consistency's sake, we recommend that you display these messages.
