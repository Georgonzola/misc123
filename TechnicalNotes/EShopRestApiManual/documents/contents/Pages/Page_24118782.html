2. Communication With the Nintendo eShop Server

This chapter describes the communication that takes place between the Nintendo eShop (referred to as "eShop" below) server and the independent server. For information about communication between the client and the independent server, see Communication With Clients.

2.1. Overview

2.1.1. How to Use the eShop REST API

The eShop REST API is provided as a Representational State Transfer (REST) API. Properties such as title information, service item information, and user information are expressed as resources. To get a resource's information, send an HTTP GET request to an URL and include the resource name and a resource identifier.

To execute service item purchases and other non-read functions, send an HTTP POST request. When doing so, set the request parameters in the body. Percent-encode the request as described in RFC 3986. Only UTF-8 can be used for the character encoding.

Note:

When sending a request, the Content-Type header value must be set as shown below.

application/x-www-form-urlencoded; charset=UTF-8

2.1.2. eShop REST API Resources

The eShop REST API provides the following resources (shown as a hierarchical structure).

  • https://akindo-(server_env).(platform_name).shop.nintendo.net/akindo/rs
    • Accounts (/accounts)
      • Balance (/balance)
      • Rights (/rights)
      • Transactions (/transactions)
    • Titles (/titles)
      • Items (/items)
        • Price (/price)

The hostname depends on the server environment name and the client platform name.

Replace (server_env) with the server environment name. For more information, see 2.1.14. Connecting to the Development and Production Environments.

Replace (platform_name) with the client platform name. Use ctr if the client is part of the 3DS family, and wup if the client is a Wii U.

Note:

The hostname and context root may change.

2.1.3. Authentication

2.1.3.1. Authentication Between the Independent Server and eShop REST API

To use the eShop REST API from an independent server, perform the following two-factor authentication.

  • Independent Server Authentication
    • Use SSL with client authentication to authenticate the independent server. Authenticate the independent server and encrypt the data.
    • To generate a client certificate, generate a key pair and submit the PEM-formatted certificate signing request (CSR) to the Online Title Management Server (OMAS). Use RSA public-key encryption with a key length of 2048 bits.
    • Use a unique proper noun as the certificate subject (common name), such as the title or the publisher. (Avoid names that are likely to overlap with other names, such as "Test cert.")
    • The certificate authority operated by Nintendo will sign and issue a certificate for the CSR you submitted to OMAS. This certificate will be released on OMAS. Use it to configure your independent server.
    • When Nintendo carries out Lotcheck, the independent server's client certificate used must be compatible with environment names that include service tokens received from the application. For information on environment name and client certificate support, see 2.1.14. Connecting to the Development and Production Environments.
  • Title Authentication
    • Independent servers can host services for multiple titles, so titles must be authenticated to prevent other title information (such as item lists) from being accessed inadvertently.
    • Use basic authentication with title codes and TINs to authenticate a title. Use the title code as the username and the TIN as the password to generate an authentication header, and then set this header in all requests, using the following example.
      • Example:
        Host: https://akindo-d1.ctr.shop.nintendo.net/akindo/rs
        Authorization: Basic [authentication header generated from title code and TIN]
    • The eShop REST API returns HTTP/1.1 401 Authorization Required if the authentication header is not set or if the title code and TIN combination is invalid.
    • For more information about title codes, see the title code description. You can access TINs from the relevant title pages on OMAS.

This two-factor authentication is applied to all requests made to the eShop server.

Note:

When submitting the CSR to OMAS, set different common names for the development and production units. Contact Nintendo if you would like to use the same common name for both development and production units.

Note:

Even if a certificate for submitting CSR to OMAS has been issued, Nintendo must configure the eShop server, so the server may not be immediately available for use. Also, a production CSR can be submitted before Lotcheck has been performed.

2.1.3.2. Authentication Between the Game Application and the Independent Server

A service token acquired from the account server is included in communication between the game application and the independent server, therefore make sure to encrypt the communication using SSL. Use the CA of your choice for signing the certificate. We recommend RSA 2048-bit public-key encryption.

Account IDs and principal IDs on different account servers may conflict. To identify a Nintendo network account on an independent server, be sure to use a unique ID based on a combination of the account server environment name and the principal ID. Do not use the principal ID only. For more information, see the CTR Account System Developer's Guide.

Note:

Examples of the account server environments are listed below.

  • D1 through D9: Development environment
  • L1 through L9: Production environment
  • S1 through S9: Lotcheck environment

The environment name is a combination of one uppercase letter (A-Z) and one number (0-9). For example: L1. Make sure that an application can support any possible combination without issue.

2.1.4. Service Tokens

To use the eShop REST API, depending on the use, obtain one of the following two service tokens from the account server.

Independent Server Service Token

The functions provided by the eShop REST API require a principal ID to identify the user.You can get the principal ID by decrypting the service token obtained from the account server on the independent server.

To prevent spoofing and other security risks, be sure to get the principal ID from the service token. For more information about decrypting service tokens and what information they contain, see the CTR Account System Developer's Guide.

Service tokens do not expire and there are no limits on the number of times they can be used, but you can configure an expiration in accordance with the independent server's policy. If an expiration is configured, check for an expiration based on the token date of issue contained in the service token. If your independent server consists of more than one server machine, you can also share tokens between servers.

EC Service Token

Some of the functions in the eShop REST API require an EC service token. When you call one of these functions, you must get an EC service token from the dedicated function in the account library API and send it to the independent server.

Do not cache the EC service token in the independent server. Obtain a new one from the account server every time one of these functions is called.

2.1.5. Restricting the Volume of API Requests

Although there is currently no explicit limit on the volume of calls that can be made to the API, whenever possible, avoid practices such as retrieving license information periodically after the user starts the game.
We recommend that you only retrieve license information when the user uses an item or service (that is, when it is necessary to check for a license).

2.1.6. Timeout Period

Each of the eShop REST API functions times out and returns a '502 Bad Gateway' error if it cannot return a response within 150 seconds from the time the request is received.
However, because the actual timeout error is not strictly returned after 150 seconds, configure the independent server to check for timeouts after a slightly longer period. For example, approx. 180 seconds.

2.1.7. Functionality When the eShop Is Under Maintenance

2.1.7.1. API Functions Available During Maintenance

Part of the eShop REST API is not available while the eShop is under maintenance. The following functions are available during maintenance.

  • Listing Items
  • Getting Account Balance
  • Getting License Information
  • Getting Purchase History

If trying to execute any other function during maintenance, an error code is returned indicating that the eShop is under maintenance.

2.1.7.2. Connecting to a Different Server During Maintenance

While the eShop is under maintenance, the IP address of the eShop REST API changes.

In this documentation, the server connected to during non-maintenance periods is referred to as the primary, and the server connected to during maintenance is referred to as the secondary. For example, during eShop maintenance, the destination REST API switches to the secondary.

The eShop switches to the secondary as follows.

Figure 2-1 Steps for Switching Servers During Maintenance

When also connected to the secondary, the server returns an error code if any API function is called that is other than those in 2.1.7.1. API Functions Available During Maintenance.

The switch to the new destination IP address is made by the public DNS provided by Nintendo, which manages the nintendo.net zone.
Use a full-service resolver (personal or one provided by your ISP) to connect to the eShop REST API.

There is a delay of at least 10 minutes between executing steps 2 and 3. However, we recommend setting the DNS cache expiration in accordance with the TTL specified by the DNS provided by Nintendo.

After maintenance ends, the eShop switches back to the primary as follows.

  1. Maintenance ends.
  2. The DNS that manages the nintendo.net zone returns the primary IP address for akindo-(server_env).(platform_name).shop.nintendo.net.
  3. All API functions can be called on the primary.

2.1.8. Title Code

The eShop REST API uses the title code as the game title identification ID.

For example, when listing items, specify the title code to get a list of items related to that title. The title identifier included in the authentication header, when authenticating titles, also uses this title code.

Wii U Titles

Specify the title codes for Wii U titles in the following format.

1 + [unique ID] + 00 (8-digit hexadecimal)

For example, with a unique ID of D0001, the title code would be 1D000100.

3DS Titles

Specify the title codes for 3DS titles in the following format.

0 + [unique ID] + 00 (8-digit hexadecimal)

For example, with a unique ID of D0001, the title code would be 0D000100.

2.1.9. Time Zone of the Time Returned by the Server

Times returned by the server are in the UTC time zone, and follow the ISO 8601 format. (For example, 2012-06-05T11:22:33Z.)

2.1.10. Character Encoding Used by the eShop REST API

The character codes for all eShop REST API requests are handled in 8-bit UCS (Unicode) Transformation Format (UTF-8). All responses are handled the same way.

2.1.11. Currency Codes Used by the eShop REST API

The eShop REST API uses the currency codes stipulated by ISO 4217.

2.1.12. Language Codes Used by the eShop REST API

The eShop REST API uses the 2-letter language codes stipulated by ISO 639.

2.1.13. Rules for Language Use

All currency denominations in responses from the API are formatted according to the language. If you specify a language code in a call to the API, the response is formatted in accordance with that language.

Conversely, if you omit the language code, the response is formatted for the language used by the user to access Nintendo eShop. Note that if a country supports more than one language, the language in the system settings may not always match with the language used to access Nintendo eShop.

In general, we recommend passing the user's system settings language as a parameter to any call of an API function.

2.1.14. Connecting to the Development and Production Environments

The development, Lotcheck, and production environments are separated by their FQDN.

The FQDN must include the server environment name and the platform name, as indicated below. 

akindo-(server_env).(platform_name).shop.nintendo.net

For the server environment name, use the value obtained when decrypting the service token acquired from the account server. For the platform name, use ctr if the client is part of the 3DS family, and wup if the client is a Wii U.

Examples for each environment are provided below.

  • Development Environment (Access is allowed with client certificates for development.)
    • akindo-d1.ctr.shop.nintendo.net
    • akindo-d1.wup.shop.nintendo.net
  • Lotcheck Environment (Access is allowed to s1 through s3 with client certificates for development, and to s4 through s6 and d9 with client certificates for production.)
    • akindo-s1.ctr.shop.nintendo.net
    • akindo-s1.wup.shop.nintendo.net
    • akindo-s4.ctr.shop.nintendo.net
    • akindo-s4.wup.shop.nintendo.net
    • akindo-d9.wup.shop.nintendo.net
  • Production Environment (Access is allowed with client certificates for production.)
    • akindo-l1.ctr.shop.nintendo.net
    • akindo-l1.wup.shop.nintendo.net
Note:

The North American and European Lotchecks are divided into two stages.
Use the s1 to s3 server environments for the first stage, and the s4 to s6 server environments for the second stage.
When performing the independent server authentication described in 2.1.3.1. Authentication Between the Independent Server and eShop REST API, switch the client certificate to be used by basing it on the server environment name.

Note:

A title that already use a FQDN that differs from the ones described previously can continue to use that FQDN.

Note:

The Lotcheck environment cannot be used until Lotcheck begins.

2.1.15. License Types

The eShop REST API can issue the following three types of licenses (called tickets).

Table 2-1. License Types and Descriptions
  License Type     Descriptor     Description  
Subscription Ticket ADDITIONAL TIME Expresses the time until the item expires in epoch seconds (seconds since 00:00:00 UTC, January 1, 1970).
Consumable Ticket CONSUMPTION Expresses the number of times that the item can be used. The independent server must manage the number of uses that have been consumed.
Permanent Ticket PERMANENT Expresses a license that can be permanently used with the item.

2.1.16. Response

2.1.16.1. Response Format

The eShop REST API provides XML and JSON response formats. Specify the response format by specifying a content type in the Accept header of your request.

To receive responses in JSON format, you can use either application/nintendo-eshop-format-v2.0+json or application/json. application/json is the old format. The current recommendation is to use application/nintendo-eshop-format-v2.0+json.

Table 2-2. Response Formats
Accept Header Value in Request Content Type of Response
None application/xml
application/xml application/xml
application/nintendo-eshop-format-v2.0+json application/json
Note:

If the response format is set to the old JSON format, elements composed entirely of numerical values, such as raw_value, will not be returned in double quotation marks. Therefore, when results containing a decimal point are returned, they are handled using a data type such as a float value, and if the value is included in a request, it may result in an error.

On independent servers, ensure that issues due to use of an unintended data type do not occur. For example, handling values as strings.

2.1.16.2. Root Element of the Response

The root element of all eShop REST API responses is <eshop>.

2.1.16.3. Version

The version number of the eShop API is included in all eShop REST API responses.

Code 2-1. Version Example
0001  <eshop>
0002    <version>1.0</version> 
0003    :
0004  </eshop>

2.1.16.4. Reference ID

The reference ID is an ID for uniquely identifying an item for sale. It is defined on the independent server side. Use 32 hexadecimal characters for the reference ID.

The reference ID returned in the response from the eShop REST API is entirely in lowercase letters. Note that the reference ID in the response is entirely in lowercase even if it was set using uppercase in the request.

2.1.16.5. Errors

If an eShop REST API function fails, it returns error information as the response. Error information consists of an error code.

Although the implementation of error handling is generally handled by the independent server, if an error code is listed in the e-commerce guidelines provided by Nintendo, we ask that you display it in the client application using the error viewer.

Error Code Format

030-xxxx (where "xxxx" is a 4-digit decimal number)

Code 2-2. Example of Error Information
0001  <error>
0002    <code>030-1111</code>
0003  </error>

2.1.16.6. Correlation ID

A unique correlation ID is included in all eShop API responses. If errors occur, include this ID when contacting Nintendo with any inquiries.

Code 2-3. Example of Correlation ID
0001  <correlation_id>cor112358132134</correlation_id>
Note:

To check the detailed settlement information, contact Nintendo with the PID and ext_transaction_id.

2.2.1. Basic API

2.2.1.1. Listing Items

Overview

Gets a list of subscription tickets or consumable items defined by the game title that has the specified title code.

The independent server then generates the item list that is ultimately shown to the user, using the item codes to match items on the item list maintained on the independent server with the items in the item list obtained by using this API.

If the number of items is large, you can restrict the number of items obtained by specifying an offset value and the maximum number in the list. The items obtained in the item list are only those items currently sold in the country that is set for the specified Nintendo Network account. Items that are planned for sale, or for which sales have already ended, are not included in the list.

2.2.1.3. Getting Pre-Purchase Information

Overview

You can get information about the item specified by the item code before purchase. The kind of information available before purchase includes the item's retail price, the current balance, and the balance after purchase.

The retail price is the item's price as of the receipt of the request.

The tax amount appears if the Nintendo Network account's country setting is for a tax-exclusive country. The tax amount is calculated based on the region of the specified Nintendo Network account.

Note:

If the post-purchase balance is negative, the purchase process always fails. Do not call the API that requests a purchase.

Note:

It is not possible to add to a balance using the eShop REST API. Funds must be added from the eShop application.

Note:

On 3DS, you can send a request to the EC applet for the addition of the balance shortfall.

You can use this API to determine whether there are sufficient funds by checking if the post-purchase balance is negative.

If there are insufficient funds, the post-purchase balance string without the negative sign (-) is the amount of the shortfall. When adding funds via the EC applet, only add the amount that the balance was short.

2.2.1.4. Purchasing Items

Overview

Purchases the specified item. If the purchase is successful, a license for the relevant item is granted to the specified Nintendo Network account.

In the process of making a purchase, the application sends the independent server a purchase request, which the independent server then sends to the eShop Server. The eShop Server then responds with the results of the purchasing process, which the application should display as appropriate.

When communicating with an independent server, the independent server's service token and EC service token are required. The application must get an EC service token from the account server, and then send that token 'as is' to the eShop Server. One EC service token must be obtained for each purchase request.

Get an EC service token using the dedicated account library API. EC service tokens are only valid for one minute after being issued. Do not cache them in the independent server. A new one from the account server must be obtained for every purchase.

Note:

It is not possible to add to a balance using the eShop REST API. Funds must be added from the eShop application or the EC applet.

2.2.1.5. Getting License Information

Overview

Gets the license information for the specified service title.

This can be used to retrieve the license information for all of the Nintendo Network accounts linked to devices that are tied to the specified Nintendo Network account. Even if the specified Nintendo Network account does not have a license, if another Nintendo Network account does have a license, the system returns the license information for that network account.

In accordance with the ticket type, process the obtained ticket information as follows.

Permanent ticket

For permanent tickets, provide the service as soon as the ticket information from the eShop Server is received.

Subscription ticket

For subscription tickets, the independent server must first check that the ticket information is currently valid and has not expired.

Ticket information obtained from the eShop Server contains the valid period as a UNIX® system timestamp value. Compare the valid period time against the current time, and provide the service if the ticket is currently valid. For the current time, use the independent server's time synchronized against an NTP, or the eShop Server time returned together with the ticket information.

Consumable service ticket

For consumable service tickets, the information obtainable from the eShop Server only gives the total number of ticket usages that have been purchased. Note that this value is not the number of times remaining that the ticket can be used.

Store the number of times the ticket has already been used on the independent server, and then use that number to calculate the number of times remaining that the ticket can be used. Subtract the total number of times the ticket has already been used, as stored on the independent server, from the total number of ticket usages purchased, as stored on the eShop Server, and only provide the service for the remaining number of times the ticket can be used.

An example of where this would be relevant would be a power-up item sold for use in an online action game. Assuming the user purchases three power-up items, the ticket information obtained from the eShop Server would record three total ticket usages purchased. Once the user has used one of the three power-up items, record on the independent server that the ticket has been used one time. The difference between the three total ticket usages and the one time the ticket has been used would leave two remaining times that the ticket can be used, and the application would inform the user of this.

2.2.1.7. Getting the Payment Status

Overview

Get the payment status for a specified purchase from the purchase history of a particular network account.

Gets the payment status for each title code. Only the payment status for the specified network account can be obtained. This differs from the license information retrieval API.

Using the API

The API for getting payment status is for use when user item license information is managed independently by an independent server, and the Nintendo eShop server does not get and confirm the license information.

This type of implementation is restricted by the guidelines. However, when using the eShop mechanisms to provide a time-based service, the following constraints apply, so the guidelines may be ignored in some cases but after advance consultation with Nintendo.

  • Period of validity could start at some arbitrary time after purchase.
  • If a period of time occurs when the user cannot use the service due to maintenance of an independent server or an other unforeseen issue, it is not possible to compensate the user (extend validity period) from just the independent server.

When license information is managed only on an independent server, inconsistencies can arise between license information at the eShop and on the independent server. This can happen in cases such as the following.

  • A timeout occurs internally on the shop server during the purchase.
    • An error is returned as the response from the front-end shop server to the independent server.
    • However, in some cases, the processing on the server performing the purchase itself completes and the amount is deducted from the user's account.

When a state occurs as described above, if the mechanism adds to the user's license information based on a successful purchase response at the independent server, then the license may not be granted even though the purchase was completed.

This API should be used to resolve inconsistencies arising between the eShop server and independent servers when cases as described above occur.

Specifically, use it as follows.

  • If a purchase request fails due to an internal server error, check whether the purchase in the transaction completed or not.
  • Periodically batch process transactions that had errors and correct any inconsistencies.
Note:

This API creates load on the eShop server, so it should be used only in fault conditions (when a purchase fails). It must not be called for normal cases, such as when purchase completed successfully.

Note:

To use this API in any other type of case, first contact Nintendo.

Restriction on the Number of API Calls

A limit on the number of calls to the payment status API has been set.

Currently a limit of 1800 calls per 900 seconds is permitted, and an error will be returned if this limit is exceeded.

2.2.2. API for Recurring Billing

2.2.2.1. Obtain List of Recurring Billing Items

Overview

Gets a list of recurring billing items defined by the game title that has the specified title code.

Items must be registered beforehand as eShop backend server processing automatically occurs during each payment cycle. Hence, the list of items displayed then is different from the usual service item list, and matching with the item list saved on the independent server is not required.

If the number of items is large, you can restrict the number of items obtained by specifying an offset value and the maximum number in the list.
The items obtained in the item list are only those items currently sold in the country that is set for the specified network account.
Items that are planned for sale, or for which sales have already ended, are not included in the list.

Request

Table 2-15. Parameters
Parameter Optional Description
title_code No Title code identifying the game title (8-digit hexadecimal).
pid No Principal ID for the network account. Use the value obtained from the service token received from the client.
lang Yes

Language configured in System Settings. Use the language code received from the client. Used to determine the currency display format.

It is also used to determine the display language for usage records and receipts. If omitted, the system uses the eShop language for the specified network account.
offset Yes Offset for getting service items from the search results that match the filter.
limit Yes Number of service items to get as search results.
Code 2-29. URL
0001  GET https://akindo(-server_env).platform_name.shop.nintendo.net/akindo/rs/titles/ $title_code /auto_billing?lang= $lang & pid= $pid & offset= $offset & limit= $limit

2.2.2.2. Getting License Information that Includes Recurring Billing Contract Information

Overview

Gets the license information for the specified service title.

This can be used to retrieve the license information for all of the network accounts linked to devices that are tied to the specified network account. Even if the specified network account does not have a license, if another network account does have a license, the system returns the license information for that network account.

If recurring billing is set in the title, it also returns whether the consecutive purchase of subscription tickets occurs automatically.

Request

Table 2-16. Parameters
Parameter Optional Description
pid No ID to identify the Nintendo Network account. Obtain by decoding the service token received from the client.
title_code No Title code identifying the game title (8-digit hexadecimal).
reference_id Yes Reference ID (32 hexadecimal characters). Lowercase alphanumeric [a-f0-9]{32}. If a reference ID is specified, returns only the license information for the relevant reference ID. If not specified, returns the license information for all of the reference IDs for the title.
