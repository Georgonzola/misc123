3. Communication With Clients

This chapter describes the communication that takes place between clients and an independent server, using an independent server demo application as an example. For information on communication between the eShop server and an independent server, see 2. Communication with Nintendo eShop Server.

3.1. Overview

3.1.1. Overview of the Independent Server Demo Application

The independent server demo application simulates an independent server being run by a service provider.

The application combines the item information managed on the independent server with the item information retrieved from the eShop REST API to create a list of purchasable items and return it to the client.

The client can purchase items selected from the item list and use any licenses that were purchased for available items. The application also provides a database management interface for manipulating the item information and user information managed by the independent server.

3.1.2. Configuring the Independent Server Demo Application

The independent server demo application is provided as a web application that returns an HTML document using HTTP communication. It provides two features: an interface that simulates communication between a game application and an independent server, and a database management interface for the independent server.

The configuration of the independent server demo application, including the game application and eShop REST API, is illustrated below.

Figure 3-1. Configuration of the Independent Server Demo Application

For example, if the game application requests an item list, the following type of process takes place.

  1. The game application wants a list of items available for purchase from the independent server, so it sends a request including a service token.
  2. The independent server gets the title and account information from the service token it received and then, based on that information, requests the item information from the eShop REST API.
  3. The item information received from the eShop REST API is merged with the item information managed inside the independent server to create a list of items available for purchase.
  4. The created item list is then returned to the game application.

Note the following issues when actually building an independent server.

  • Have transactions between the game application and the independent server take place via service tokens.
  • For the communication interface between the game application and the independent server, we recommend sending responses via a REST API or similar means in a format that is easy for the game application to parse, such as XML, JSON, or plain text.
    (The demo application returns an HTML document under the assumption that it will be tested from the browser.)
  • You do not need to provide a database management interface for the independent server to the game application.

3.1.5. Authentication

3.1.5.1. Authentication Between the Game Application and the Independent Server

SSL encryption is required for communication between the game application and the independent server. To use encryption, configure the following two items.

  • Embed the server certificate into the independent server.
  • Within the game application, embed a certificate issued by the CA that signed the server certificate.

The following sample procedure shows how to create a self-signed certificate for development using openssl and keytool.

# 1. Create a secret key (server.key).
openssl genrsa 2048 > server.key

# 2. Create a certificate signing request (server.csr).
#  - Enter the requested information after executing the command.
#  - For Common Name, enter the independent server's FQDN.
#  - Enter this CSR in the eShop Web Service Use Application part of the independent server application on OMAS.
openssl req -new -key server.key > server.csr

# 3. Create a server certificate (server.crt).
#  - The following command sets the expiration to ten years.
openssl x509 -days 3650 -req -signkey server.key < server.csr > server.crt

# 4. Convert the server certificate created by OpenSSL (server.pfx) to pkcs12 format.
#  - After the command is executed, a password is requested.
#  - You need the password to embed the server certificate into the independent server.
openssl pkcs12 -export -out server.pfx -inkey server.key -in server.crt

# 5. Using keytool, convert the pkcs12-format server certificate to the jks format (server.keystore).
keytool -importkeystore -srckeystore server.pfx -destkeystore server.keystore -srcstoretype pkcs12 -deststoretype jks

Properly install the created server certificate on the independent server.

The following example uses the Tomcat config file used in the server demo application (%CATALINA_HOME%\conf\server.xml).

<Connector port="8443" protocol="org.apache.coyote.http11.Http11Protocol"
           maxThreads="150" SSLEnabled="true" scheme="https" secure="true"
           keystoreFile="C:\server.keystore"
           keystorePass="your_password"
           clientAuth="false" sslProtocol="TLS" />

To use SSL communication between the game application and the independent server, bundle the server certificate created in step 3 with the game application.

3.1.6. Service Tokens

In the independent server demo application, the internal server uses the eShop REST API.

To use game applications that access the independent server demo application, depending on the use, obtain one of the following two service tokens from the account server.

  • Independent Server Service Token
    • The functions provided by the eShop REST API require a principal ID to identify the user. You can get the principal ID by decrypting the service token obtained from the account server on the independent server. To prevent spoofing and other security risks, be sure to get the principal ID from the service token.
    • Service tokens do not expire, and there are no limits on the number of times they can be used. If you configure an expiration in accordance with the independent server's policy, check for expiration based on the token date of issue contained in the service token.
    • If your independent server consists of more than one server machine, you can also share tokens between servers.
  • EC Service Token
    • Some of the functions in the eShop REST API require an EC service token. When you call one of these functions, you must get an EC service token from the dedicated function in the account library API and send it to the independent server.
    • Do not cache the EC service token in the independent server. Obtain a new one from the account server every time one of these functions is called.

3.1.7. Title Code

The eShop REST API uses the title code as the game title identification ID.

For example, when listing items, specify the title code for the game title to get a list of items related to that title.

The title identifier included in the authentication header when authenticating titles is the same as this title code.

Specify the title codes for CTR titles in the following format.

0 + [unique ID] + 00 (8-digit hexadecimal)

3.2.1.2. Decrypting a Service Token

The following information, including the principal ID, can be retrieved from the decrypted service token.

  • Platform ID
    Information used to identify the platform. Fixed to 0 for the CTR.
  • Principal ID
    A constant 32-bit numeric value that is assigned to each Nintendo Network account.
  • Network account ID
    A string of up to 16 alphanumeric characters.
  • Server environment
    A string that identifies the environment of the CTR that issued the service token. (Development Hardware: D1, Retail Hardware: L1)
  • Game version
    Version information for each game, managed on the Nintendo account server. (4-digit hexadecimal)
  • Unique ID
    An ID issued by Nintendo to identify a game. This is the same as the ID that is assigned to each title. (5-digit hexadecimal)
  • Token created time
    The UNIX timestamp (milliseconds) of when the token was generated.
  • Authorization code

3.2.3. Session Information

3.2.3.1. Overview

Log into the independent server and get the session ID of the session that was created. Before using any API element other than the login API, you must first get the session ID from the login API and then include it in the Set-Cookie header (for example, "JSESSIONID=****").

During login, get license information from the eShop REST API server and cache it. The cache can be explicitly updated.

If you logout, the session of the specified session ID is destroyed. Afterward, it will no longer be possible to send any requests or responses using that session ID.

Note:

The independent server demo application is implemented as a stateful web application that uses sessions.

If you want to implement the application as a stateless web application, pass an independent service token as a parameter to all API requests.

3.2.5. Show Pre-Purchase Information

3.2.5.1. Overview

You can get information about the item specified by the item code before purchase. The kind of information available before purchase includes the item's retail price, the current balance, and the balance after purchase.

The retail price is the item's price as of the receipt of the request. The tax amount appears if the Nintendo Network account's country setting is for a tax-exclusive country.

The tax amount is calculated based on the region of the specified Nintendo Network account.

3.2.7. Getting License Information

3.2.7.1. Overview

Gets the license information for the specified service title.

Gets the license information for all of the Nintendo Network accounts linked to devices that are tied to the specified Nintendo Network account. Even if the specified Nintendo Network account does not have a license, if another Nintendo Network account associated with the same device does have a license, the system returns the license information for that network account.

3.2.7.2. Request

Table 3-6. Parameters To Specify in Request
Parameter Optional Description
reference_id Yes Reference ID (32 hexadecimal characters). Lowercase alphanumeric [a-f0-9]{32}. If a reference ID is specified, returns only the license information for the relevant reference ID. If not specified, returns the license information for all of the reference IDs for the title.

URL

http://localhost:8888/v1_0/rs/accounts/own/rights/titles/authorized/items/$reference_id
Example: GET http://localhost:8888/v1_0/rs/accounts/own/rights/titles/authorized/items/0123456789abcdef0000000000000001

http://localhost:8888/v1_0/rs/accounts/own/rights/titles/authorized
Example: GET http://localhost:8888/v1_0/rs/accounts/own/rights/titles/authorized

With the Reference ID Specified

    • reference_id: Reference ID to identify the item purchased (32-character hexadecimal).

3.2.9. Show Purchase History

3.2.9.1. Overview

Gets the purchase history for the specified Nintendo Network account.

Gets the purchase history for each title code. Also, unlike with the license information retrieval API, you can only get the purchase history for the specified Nintendo Network account.

3.2.9.2. Request

Table 3-8. Parameters To Specify in Request
Parameter Optional Description
offset Yes The offset for all purchase history transactions.
limit Yes The number of purchase history transactions to get.

URL

GET http://localhost:8888/v1_0/rs/accounts/own/transactions?offset=$offset&limit=$limit
Example: http://localhost:8888/v1_0/rs/accounts/own/transactions?offset=0&limit=20

3.2.9.3. Response

  • transactions: Root node of the purchase history information.
    • total: Total number of items in the purchase history.
    • length: Number of purchase-history items in this response.
    • offset: Offset.
    • transaction: Purchase-history information.
      • transaction_id: Transaction ID.
      • ext_transaction_id: Transaction ID specified by the independent server.
      • date: Transaction date and time.
      • type: Transaction type (PURCHASE / REDEEM / DELETE).
      • description: Transaction details.
      • transaction_amount: Total amount paid, including tax.
        • amount: The total amount paid, formatted for the currency (for example, $10.00).
        • currency: The currency code defined by ISO 4217 (for example, USD).
        • raw_value: The unformatted total amount paid (for example, 10.0).
      • item: Item information (Returns null if the purchased item has been removed from the catalog.)
        • reference_id: Reference ID to identify the item purchased (32-character hexadecimal).
        • item_name: Item name of the purchased service item (for display in the purchase history).
        • item_code: Item code.
        • limit: License information for this service item.
          • type: License type for this service item.
          • value: Value corresponding to the license for this item (limited period of time, limited number of times). If the license type is unlimited, this value has no meaning.
