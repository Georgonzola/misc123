nn::act Namespace

Description

Namespace for the account library.

Overview

The account library provides functionality for retrieving information recorded in the CTR account system and for authentication based on the recorded information. The application does not need to implement any data entry for creating, editing, or authenticating accounts because these steps are handled by the system. For an overview of the CTR account system, see the CTR Account System Developer's Guide included in the SDK.

Result Handling

The Result objects returned by functions in the nn::act namespace are not detected individually. Instead, they are handled using the following technique. This example shows how to catch all errors that belong to nn::act::ResultAuthenticationError. 

            if(result <= nn::act::ResultAuthenticationError())
            {
                // Can also catch nn::act::nn::act::ResultInvalidUniqueId.
            }

Use of the == and != operators on Result objects returned by functions in the nn::act name space is prohibited and causes a build error. Note that the following two samples do the same thing.

            if(nn::act::ResultCanceled::Includes(result))
            {
                ...
            }
            if(nn::act::ResultCanceled() >= result)
            {
                ...
            }

The success and failure values returned by functions could be extended. Use the nn::Result::IsSuccess and nn::Result::IsFailure functions to determine success and failure except when handling specific results.

Errors That Must Not Occur in Retail Products

The following Result objects are caused by application bugs. You must fix the application so that these errors do not occur in a retail product.

  • nn::act::ResultNotInitialized
  • nn::act::ResultInvalidArgument
  • nn::act::ResultInvalidPointer
  • nn::act::ResultOutOfRange
  • nn::act::ResultInvalidSize
  • nn::act::ResultInvalidFormat
  • nn::act::ResultInvalidHandle
  • nn::act::ResultInvalidValue
  • nn::act::ResultOutOfUserHeap

Using API Functions Involving Communication

Observe the following practices when using functions that communicate with the account server (such asnn::act::InquireNetworkTime and nn::act::applet::AcquireIndependentServiceToken).

Requesting a Network Connection

Send a network connection request in advance using the nn::ac::CTR::Connect or nn::ac::CTR::ConnectAsync function in the automatic connection (AC) library and confirm the connection before calling any functions in the account library.

Error Handling

If a failure occurs when establishing a connection to the network, display an error message using the method for the automatic connection library. If an account library function returns an error after the network connection succeeds, use the nn::act::GetErrorCode function to get the error code and display it in the error viewer. If the program encounters nn::act::ResultCanceled, which occurs when the user explicitly cancels the authentication process, terminate the application's token acquisition sequence without displaying anything in the error display applet. If a result other than nn::act::ResultCanceled is returned, it does not need to be individually handled by the application, but the application can optionally be implemented to handle nn::act::ResultApplicationUpdateRequired and proceed to a patch download confirmation sequence.

Login Applet

When the nn::act::applet::AcquireIndependentServiceToken and nn::act::applet::AcquireOlvServiceToken functions are called, the system starts the login applet and delegates processes related to account authentication, such as password entry and token retrieval. The login applet continues to run until the completion of the account authentication and token retrieval sequence. If the application needs to continue communication with the NEX server or independent server, do so in a separate thread.

Note on Responsiveness

If a function causes communication, it is explicitly stated in the API reference. Functions that do not explicitly state this fact do not cause any communication. However, the API functions in the account library might call high priority functions that cause a heavy load on the system and adversely affect responsiveness, even if they do not cause any communication. Note that calling these functions from the main loop can result in slowdown.

Note on Synchronizing Account Information

The Nintendo Network account registration information that can be retrieved from the account library is synchronized by the system as required. This synchronization sometimes occurs while the application is running, but the information is not necessarily the latest. Due to the potential for confusion if, for example, the account ID were to change during gameplay, cache the information as appropriate in the application.

Note on Output Variable Values When Functions Fail

The values stored in output variables are not guaranteed when a function fails.

Note About Function Calls When the Library Is Uninitialized

With the exception of some functions like nn::act::Initialize, nn::act::Finalize and nn::act::IsInitialized, you must initialize the library before calling any functions. If you call a function that requires library initialization and the library is not initialized, non-Release builds stop on an assertion.

Functions

Library Initialization/Finalization
Initialize Initializes the account library and makes the account features available for use. When using APIs that trigger communication with some servers, you must set a data buffer for communication as an argument. Because the size of the required buffer varies by API, see the reference for the API you are using, and then specify the largest size from there. When you are not using these APIs, you do not need to set anything for these arguments. For example, if you are just using the feature to get a service token, you do not need to specify a communication buffer. This function can be called multiple times even if the account library is already initialized, but the nn::act::Finalize function must be called an equal number of times to completely finalize the account library. This function can only be specified when the communication buffer has not yet been initialized, however. If this function is called when the communication buffer is already initialized, the buffer specified in the argument is not used.
Finalize Finalizes the account library. If the nn::act::Initialize has been called multiple times, this function must be called the same number of times to completely finalize the account library. If this function is called more times than the nn::act::Initialize function was called, ResultNotInitialized is returned.
This call succeeds even if the library is not initialized.
IsInitialized Checks whether the account library is initialized.
Getting Account Information
IsNetworkAccount Determines whether the account is a system account.
GetCountry Gets the country setting of the system account in ISO 3166-1 alpha-2 format. Gets the country setting of the Nintendo Network account associated with the system account in ISO 3166-1 alpha-2 format. However, this function gets the value the system cached for the locale, so calling this function does not trigger communication.

Pass a pointer to a buffer at least as big as NN_ACT_COUNTRY_SIZE bytes to pCountry.

When it is not a network account, the function succeeds and a blank string is copied to pCountry.

If the function fails, the value set in the buffer is undefined.
GetAccountId Gets the account ID for the system account. Gets the account ID (Nintendo Network ID) for the Nintendo Network account associated with the system account. However, this function gets the value the system cached for the locale, so calling this function does not trigger communication.

Pass a pointer in pAccountId to a buffer at least as big as NN_ACT_ACCOUNT_ID_SIZE bytes.

When it is not a network account, the function succeeds and a blank string is copied to pAccountId.

If the function fails, the value set in the buffer is undefined.
IsOverAge Determines whether the system account user is at least the specified age. This function itself does not trigger any communication and can make this determination without an Internet connection by estimating the current date and time. However, you need to activate the network clock by calling a function that communicates with the account server in advance. In addition, you must reactivate the network clock every time the application runs.

Nintendo systems calculate the age of the user based on UTC+12 time, regardless of their country or region of residence. For users in Japan, age is calculated based on 9 P.M. of the day before the date of birth.

If you want to get the age of a user directly, please contact Nintendo support. Avoid identifying the ages of users through repeated calls to this function.
GetPrincipalId Gets the principal ID for the system account. Gets the principal ID for the Nintendo Network account associated with the system account. However, this function gets the value the system cached for the locale, so calling this function does not trigger communication. The principal ID returned by this function is different from the friend account principal ID returned by the nn::friends::CTR::GetMyPrincipalId function.
GetSimpleAddressId Gets the residence information for the system account. Gets the residence information for the Nintendo Network account associated with the system account. However, this function gets the value the system cached for the locale, so calling this function does not trigger communication. The values that can be obtained correspond to the id member in the nn::cfg::CTR::SimpleAddressId structure.
GetWiiUSimpleAddressId Gets the system account's residence information in the SimpleAddressId format used by the Wii U. Gets the residence information for the Nintendo Network account associated with the system account in the SimpleAddressId format used by the Wii U. However, this function gets the value the system cached for the locale, so calling this function does not trigger communication. SimpleAddressId is passed directly to the Wii U by the application for communication purposes; there is no need to use this function beyond when the Wii U needs to parse its meaning.
GetUuid Gets the UUID of the system account. The UUID for the system account is assigned when the account is created, and does not change until the system account is deleted. The ID space is separate for each title, so different titles calling this function get back different UUIDs from the same system account. Consult Nintendo support if you want to be able to identify a system account by its UUID across multiple titles.

Currently, the UUID retrieved from this function complies with the Version 1 specifications defined in RFC 4122, but the ID format could change to the extent that it does not impair the uniqueness of the ID. Be sure to handle the ID in its entirety as a single value. Do not use any individual parts of the ID as a timestamp or device identifier.

If the function fails, the value set in the buffer is undefined.
GetTimeZoneId Gets the name of the time zone set for the system account.
GetUtcOffset Gets the time difference of the time zone set for the system account.
Other
IsNetworkTimeValidated Determines whether the network clock is active. Features that use the network clock such as the nn::act::IsOverAge function are only available when this function returns true.

Always returns true if an API has at least once successfully communicated with the account server and the account library has not yet been finalized.
InquireNetworkTime Gets the time from the server, and activates and calibrates the network clock. Features that use the network clock, such as the nn::act::IsOverAge function, are available by calling this function.

When APIs that communicate with the server, such as the nn::act::applet::AcquireIndependentServiceToken function, have succeeded at least once in advance, there is no need to additionally call this function.

You cannot directly get the time with this function.
This function causes communication with the server, during which time processing is blocked.

For information about using this function, see the section on how to make API functions that trigger communication in the nn::act namespace.
Cancel Cancels communication tasks with the account server that are waiting for completion. Cancels all communication tasks with account servers that are waiting for completion that were registered by a process called by this function. Use in a different thread than the one from which the API function that communicates with the server was called.

The API function that canceled the task returns a Result and error code according to the state at that point. Note that if the cancellation is not in time, success may be returned. The failure error code that results from the cancellation does not need to be displayed in the error viewer.
GetErrorCode Gets error codes to display to the user. Pass the value acquired from this function to the error viewer to display the error.

The results from each API function are handled based on nn::Result. Do not branch processes using the values obtained from this function.
GenerateUuid Generates a new UUID. If you want to assign a unique identifier to appropriately generated application data or objects, you can assign the UUID generated by this function.

The UUID that can be obtained from this function is now compliant with the version 1 specifications stipulated in RFC 4122. That said, it is still possible to make changes to the ID format within a range such that the uniqueness is not lost. Be sure to handle the ID in its entirety as a single value. Do not use any individual parts of the ID as a timestamp or device identifier.

If the function fails, the value set in the buffer is undefined.
