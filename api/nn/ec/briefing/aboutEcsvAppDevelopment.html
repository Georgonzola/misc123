Overview

Service items are the unit by which online services are sold, and are composed of information provided from two servers: the eShop server and the independent server owned by the game developer.

The game developer can limit the provision of online services based on licensing information in the service item.

This page describes the preparations required on the program side when selling service items.

Notes

  • The correct serial number must be written to the CTR system for applications that use the EC library. You can use the Config tool to check the system's serial number. Verify that the serial number matches the serial number printed on the back of the system. If the serial number is not written correctly and the printed serial number starts with EJF, the system can be repaired. Contact Nintendo. Any other serial number is for a prototype system that cannot be repaired.
  • Data titles created with the version of E-Commerce Kit for CTR-SDK 3.2 or later are not compatible with those created with E-Commerce Kit for CTR-SDK 3.1. (Compatible with version 3.2 and later.)
    Note that for this reason, the demo data titles on the server have been replaced and some of the 3.1 demo programs will not work.
    Also, for the same reason, when independent data titles are already used with 3.1 and you are replacing with SDK 3.2 or later, the data title must be rebuilt and uploaded to the server.
  • The system might freeze if the user removes the SD card while the ES applet is running for a purchase or otherwise.
    This issue has been noted in Symptoms Due to Design Limitations of Specifications as follows: [FS] Removing a card while it is being accessed may cause a fatal error or cause the system to stop operating.
    This freeze is something that applications cannot overcome, so consider this to be an error that your application cannot handle.
  • The development and production shop environments have some differences.
    For example, the production environment eShop for the Korean region does not allow credit card payments, but the development environment shows the option.

Preparations on the eShop Server Side

The game developer must register the following information on the eShop server.

  • Service item information (item code, price, sale period, license configuration)
  • Public key for Riv token verification

Any other service item information cannot be registered on the eShop server.

The item code is a string to identify the information of the service item registered on the eShop server.

The license configuration can be selected from one of the following three types.

Permanent ticket
Select when selling a service that can be used permanently once purchased.
Subscription ticket
Select when selling a license to use a service in units of time.
The determination of whether the license is valid at a particular point in time must be made by the independent server.
Consumable service ticket
Select when selling a license to use a service for a certain number of times.
The independent server must manage the number of times the license has been used.

The Riv token is a token generated on the independent server to verify the reference ID.

The public key to perform Riv token verification must be set on the eShop server.

The key requires an RSA key generated in PEM format.

The following is an example of generating a key with OpenSSL. 

openssl genrsa -out riv-private-key.pem -F4 1024
openssl rsa -in riv-private-key.pem -out riv-public-key.pem -pubout

For instructions on registering information on the eShop server, see the documentation provided separately.

Preparations on the Client Side

The game developer must implement the following features on the client.

Get information about the service item from the independent server.
The game developer must implement this feature independently.
Get information about the service item from the eShop server.
Use nn::ec::CTR::ServiceItemCatalog::Retrieve and nn::ec::CTR::ECardCatalog::Retrieve.
Get the Riv token from the independent server.
The game developer must implement this feature independently.
Purchase service items from the eShop server.
Use nn::ec::CTR::EcApplet::RequestPurchaseServiceItem or nn::ec::CTR::EcApplet::RequestRedeemServiceItem.
Get the challenge Blob from the independent server.
The game developer must implement this feature independently.
Send the ticket envelope to the independent server.
Send the ticket envelope generated with nn::ec::CTR::ServiceTitle::GetTicketEnvelope.
Request service to the independent server.
The game developer must implement this feature independently.

Communication between the client and independent server must be protected with SSL or the like.

The application is responsible for protecting communication between the client and server, and so this communication is not a category in this library.

Use the Riv token and challenge Blob immediately after getting them.

The license information for the purchased service item is registered in the ticket after making a correspondence with the reference ID.

A single ticket can have and maintain a maximum of 3000 reference IDs.

For implementation examples of each of these features, see the sample code.

Preparations on the Independent Server Side

The game developer must implement the following features on the independent server.

Incorporate the ECSV library into the independent server to be able to sell and manage service items.

Provide information about the service item to the client.
The game developer must implement this feature independently.
Send the Riv token to the client.
Send the Riv token generated by the EC_GenerateRIVToken function.
Send the challenge Blob to the client.
Send the challenge Blob generated by the EC_GetChallenge function.
Verify the ticket envelope received from the client.
Use EC_VerifyExtendedTicket.
Verify license information for the service item specified from the client.
The game developer must implement this feature independently.
Provide the service to the client
The game developer must implement this feature independently.

Communication between the client and independent server must be protected with SSL or the like.

The application is responsible for protecting communication between the client and server, and so this communication is not a category in this library.

The information for the service item provided from an independent server must include the following.

Item code.
Specify the item code the indicates the corresponding information on the eShop server.
Service name
Name of the service item. There are no restrictions regarding settings on the server, but when displaying the service name using the EC applet on the client side, it can only be a maximum of 32 characters, including the NULL terminator.
Reference ID.
A 16-byte binary string to identify the service. You can freely specify this string.

The item code and reference ID pair have a one-to-one correspondence with a single service item.

You can assign multiple item codes to a single reference ID or, conversely, assign multiple reference IDs to a single item code. However, the reference ID and license configuration must have a one-to-one correspondence.

To generate a Riv token, you must have a private key that corresponds to the public key for verification of the Riv token registered on the eShop server.

Generate a new Riv token and challenge Blob for each communication channel.

For implementation examples of each of these features, see the sample code.

Using OpenSSL With Independent Servers

The ECSV library uses features of OpenSSL.

But depending on the version of OpenSSL being used, the ECSV library may not operate correctly. To check whether the ECSV library can be used with the version of OpenSSL that you are using, use the following procedure.

  1. Using this command, extract the terminating 1180 bytes from the file resources/ecsv/cert_dpki.sys included in the CTR-SDK.
    • tail --bytes 1180 cert_dpki.sys > cert_ec-pub-key.txt
  2. Execute this command.
    • openssl x509 -in cert_ec-pub-key.txt -noout -text

If the result parses correctly as in the following sample, you can use the ECSV library. 

             Public Key Algorithm: id-ecPublicKey
                        Public-Key: (233 bit)
                        pub: 
                            04:00:e4:84:23:ac:7e:49:e3:69:5d:61:6c:6f:f0:

The output is similar to the following if you are using a version of OpenSSL that the ECSV library does not support. 

        Subject Public Key Info:
                    Public Key Algorithm: id-ecPublicKey
                                Unable to load Public Key
                                140204109485896:error:100AE081:elliptic curve routines:EC_GROUP_new_by_curve_name:unknown group:ec_curve.c:316:
                                140204109485896:error:100D7010:elliptic curve routines:ECKEY_PUB_DECODE:EC lib:ec_ameth.c:206:

Debugging

Game developers are asked to use the following procedures to confirm that user rights information is correctly tracking to what is registered in the Nintendo Network ID.

To run these procedures, you must use the System Updater that supports the registering of Nintendo Network IDs.

Use the following procedure when an application does not require the registration of a Nintendo Network ID for the use of a service item.

  1. From System Settings, register a Nintendo Network ID.
  2. Purchase a service item.
  3. Confirm that you can use the service item purchased from within the application.
  4. From System Settings, format the system.
  5. Confirm that you cannot use the service item from within the application.
  6. From the System Settings, reassociate the same Nintendo Network ID as in step 1.
  7. Confirm that you can use the service item from within the application.
  8. In System Settings, in Nintendo Network ID Settings, select Delete Nintendo Network ID, and delete the Nintendo Network ID.
  9. Confirm that you cannot use the service item after starting the EC applet.

Use the following procedure when an application requires the registration of a Nintendo Network ID for the use of a service item.

  1. From System Settings, register a Nintendo Network ID.
  2. Purchase a service item.
  3. Confirm that you can use the service item purchased from within the application.
  4. From System Settings, format the system.
  5. From System Settings, register a Nintendo Network ID that is different from the one registered in step 1.
  6. Confirm that you cannot use the service item from within the application.
  7. Once again, format the system, and from the System Settings, reassociate the same Nintendo Network ID as in step 1.
  8. Confirm that you can use the service item from within the application.
  9. In System Settings, in Nintendo Network ID Settings, select Delete Nintendo Network ID, and delete the Nintendo Network ID.
  10. Confirm that you cannot use the service item after starting the EC applet.

Note:

If the October 2014 system update has not been applied, the behavior differs depending on whether the following actions are taken between step 4 and step 6.

  • Call the nn::ec::EcApplet::RequestInitializeSession function from the application.
  • Add to the balance with EcDevTool.

If neither of these steps is taken, the license information for the service items remains on the system and the items are available at step 6 even if a different Nintendo Network ID is registered.

If either of these steps is performed, the rights information on the system is synced with the server, so the service item becomes unavailable in step 6.

If, on the other hand, the October 2014 system update has been applied, you do not need to take either action because the license information for service items saved to the 3DS is deleted when the system is initialized.

Notes to Consider for Independent Server Specifications and When Submitting Masters

As called for in the specifications for the ECSV library, when placed on independent servers, you must switch the settings between development and production hardware. (In practical terms, the files for decryption and the environment variables are changed. Details are in the unpublished ECommerceServiceKit Readme.)

As a result, we recommend that you have two types of independent servers: development and production. Doing so enables you to continue development work after a release, and enables you to check the functionality of development hardware even when Nintendo is using the production hardware to perform Lotcheck after you submitted your master.

The following are examples of the work you can do in such circumstances.

  • Use a DNS server to complete an environment where development hardware can connect to independent servers for development.
  • To establish valid SSL communications, install a server certificate meant for production on a development-use independent server.

Compatibility When Using Different eShop Accounts

Design and implement your application and independent server to prevent problems with compatibility when referencing the same save data from different eShop accounts (as per the account ID obtained with the nn::ec::CTR::GetAccountId function, or the accountId member of the ECExtendedTicketInfo structure that you can get using the EC_VerifyExtendedTicket function of the ECSV library).

Future plans are to change the specification such that when a Nintendo Network ID is deleted or the Nintendo eShop account activity is deleted, the application's save data will not be deleted but instead will remain available to use if an application is later repurchased.

Now as in the past, when an eShop account is deleted, both the downloaded application and its save data are deleted. In a future update, when a deleted application is repurchased,the previous save data will be inherited so that it can continue to be used.

As for applications that have card-based versions, there is already a path to using the save data from a different eShop account by inserting the 3DS Card in another 3DS system.

Syncing the License Information for Consumable Service Items When eShop Account Changes

When the mechanism of the ECSV library is used for consumable service items, the number of purchased items is stored in the 3DS in the same way as save data.

On the other hand, the number of consumed items is managed by the independent server. So when the independent server resets the number of consumed items to zero due to a change in the eShop account, the information about the number of purchased items must also be reset on the 3DS system. If not, the user might be able to reuse licenses that have already been consumed.

If you are storing service-unique account information in save data, or using the principal ID of the Nintendo Network ID as a key for managing license information on the independent server, devise some mechanism for detecting a change to the eShop account and syncing the license information.

To sync license information, call the nn::ec::CTR::EcApplet::RequestInitializeSession function.

The following example shows a change to the Nintendo Network ID.

  1. Synchronize the license information before creating save data for the first time, and store the principal ID of the Nintendo Network ID obtained using the nn::act::GetPrincipalId function to this save data.
  2. If the value for the principal ID stored in the save data differs from the value obtained using the nn::act::GetPrincipalId function, call the nn::ec::CTR::EcApplet::RequestInitializeSession function to synchronize the license information, and then store the value of the nn::act::GetPrincipalId function in the save data.

This call to the nn::ec::CTR::EcApplet::RequestInitializeSession function does not violate the guideline to not connect to the eShop server when using a service item.

Note: This corresponds to Prohibition Against Connecting to Nintendo eShop Server When Using a Service in the 3DS Guidelines.

Notes on Using Nintendo Network Accounts

If a Nintendo Network ID has been registered to a 3DS system and that system is later initialized or the NNID is deleted, the fate of the license information for service items will differ, depending on whether the October 2014 system update has been applied.

  • For a 3DS system to which the October 2014 system update has not been applied:
    • The license information for service items remains, even if the system is initialized or the NNID is deleted.
  • For a 3DS system to which the October 2014 system update has been applied:
    • The license information for service items is deleted when the system is initialized or the NNID is deleted.

For more information about downloadable content rights data behavior, see ECDK: Developing Applications That Support Downloadable Content.

Displaying Account Balances

There is a possibility of the account balance changing even while the application is running, for example if the Theme Shop or another device sharing the same Nintendo Network ID is used.

When purchases are made, the correct balance is shown in the EC applet, so the application does not need to show the balance all the time. However, note that the balance can change this way if you plan to show the balance in your application.

Displaying Legal/Disclaimer Information

When showing the service legal/disclaimer information by using ECSV functionality, set the information in the nn::ec::CTR::ServiceDescription structure, and pass this as an argument when requesting a purchase to the EC applet.

This display will be limited to a maximum of 1024 characters, including NULL characters when displaying the legal/disclaimer in the EC applet.

When releasing an application in multiple languages and regions, localization may result in this limit being exceeded.

The legal/disclaimer information display may be done in the application, instead of the EC applet. We recommend considering implementing the legal/disclaimer information display in the application when this 1024 character limit is in risk of being exceeded.
