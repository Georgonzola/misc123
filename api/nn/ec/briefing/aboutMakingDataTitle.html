Overview

This section describes how to create data titles that store add-on content and how to handle them during development.

Add-On Content and Data Titles

One application can provide multiple add-on content pieces.

The CTR system treats multiple add-on content pieces provided by an application as one data title. The final step is the creation of a data title as a CIA file.

About Data Titles

Structure of Data Titles

One data title contains one metadata file, and one or more add-on content pieces.

Metadata content contains the content name, content description, and an icon image to indicate the content. The content of add-on content is maintained as a file archive. Metadata content and add-on content are created as CFA files.

Required Information for Creating Data Titles

To create a data title, information other than the add-on content data itself, such as the name of the add-on content and an icon, is required to indicate the type of add-on content.

You need the following data to create a data title containing add-on content.

Title name
Required to create the CFA file.
Unique ID
ID to identify the data title. Use the value issued by Nintendo.
Product code
Code to indicate the product. Use the value issued by Nintendo.
Data title version
Starts from 0. Increment the value by one with each release of a new version of the data title.
Icon data
Use icon data created from the BSF file. Specifically, use the ICN file.

Required Information for Creating Add-On Content

To create individual pieces of add-on content, you must have add-on content data in addition to the information required for creating a data title.

The add-on content data is a file archive in the same format as the application ROMFS, and is created as a CFA file. For this reason, the add-on content itself is placed as files in a single directory during development.

Required Information for Creating Metadata Content

Metadata content stores content attributes for all content data stored in the data title and icon image. For more information, see ctr_makeMetaDataContent.

The icon image stored in the metadata content is different from the icon image required to create data titles. The icon image stored in the metadata content indicates the individual add-on content pieces; for example, icons that indicate what the add-on content contains, such as swords or shields. The icon image required to create data titles usually represents the data title similar to an icon image that represents an application.

Data Title Updates

When a data title is updated with the CTR system, add-on content can be updated or added.

Because data titles are updated separately from the application, it is important to test updating data titles during the application development phase.

The data title update process is performed by the EC library. The data title version is used to determine whether an update is required. For this reason, when testing data title updates during application development, you must specify a version with a value greater than the original data title so that the system can identify the updated data title as being new.

Data Title Versions

The data title version rules are different from those for application versions. When developing data titles, only one value called the data title version is used.

Data title versions begin with 0, and are increased by 1 each time you submit to Lotcheck.

The data title version can be up to a maximum of 4095.

Icon Data

As with applications, data titles display on the Add-on Content Deletion screen of the Management screen, and require an icon image along with the title name and publisher name in several languages. For basic information about creating icons, see ctr_makebanner.

The created icon data is used when creating the metadata content CFA file and is included in the metadata content.

However, the icon data required for the data title differs from the application in several respects.

  • Sound and 3D models required to create banner files are unnecessary.
  • Information about ratings and regions is unnecessary.
  • Attributes for application execution, such as save data information, are unnecessary.

Required data title attributes set in the BSF file are as follows.

  • Icon Image (LittleIconFile, BigIconFile)
  • Title Name (XXXXLongName, XXXXShortName)
  • Publisher Name (XXXXPublisher)

Set this information to match the region and language where the add-on content is to be provided.

Content Indices

Content attributes in the metadata content and add-on content CFA files are associated by a content index.

The content index specified in the content attributes in the metadata content is written in an XML file used to input ctr_makeMetaDataContent.

For add-on content CFA files, the content index is specified when creating a CIA file with ctr_makecia, and is specified with a ctr_makecia program argument, or with a content list file.

Specify a content index of 0 for metadata content; use a content index of 1 or greater for add-on content.

Specify values for content indices as a sequential series beginning with 0. For example, you cannot specify a content index series such as 0,1,2,10,11.

Upper Limits on the Number of Content Pieces and Content Indices

There is an upper limit on the number of content pieces and content indices. Content exceeding this number cannot be handled. For more information about this upper limit, see CTR-ECDK Overview.

Creating and Using Multiple Data Titles

If you have more content pieces than can be placed in a single data title, or otherwise have a huge amount of add-on content to provide, you can create and use multiple data titles.

To create and use multiple data titles for a single application, you can specify a value called the data title variation. By specifying the same value for the unique ID but different values for the data title variation, you can create multiple data titles.

The data title variation can take values between 0 and 255. If only one data title will be used, specify 0 for the data title variation.

Add-on content provides a feature so that content can be added and updated after the application has been released. However, it provides no mechanism for adding data titles. For this reason, when developing an application that uses add-on content, you must estimate whether a single data title will be enough and decide whether to use multiple data titles. Before releasing the application, you must test the updating of add-on content to multiple data titles.

If you want to make use of multiple data titles, please contact Nintendo.

How to Create Data Titles

Both the method for writing an OMakefile, and the method for directly using the tools can be used to create data titles.

When building using an OMakefile, you can create a data title by writing only minimal settings. However, this method cannot be used when including a data title in an independent build system or when creating a data title that stores a large volume of add-on content that exceeds several hundred pieces. For this reason, depending on the data title to be created, directly using the tools to build may be required without using the existing build rules that can be used by writing an OMakefile.

Build Using OMakefile

A data title can be created using OMakefile by specifying information with an OMakefile. For the information to specify, see SampleDemos/ec/DataTitle.

Variable Name Description
TARGET_AOC_TITLE Specifies the application title in eight or fewer ASCII characters.
UNIQUE_ID Specifies a unique ID.
PRODUCT_CODE When you are building add-on content, specify CTR-M-XXXX-00.
M is specified for the configuration code.
The initial code of the application using the add-on content is specified for XXXX. The 00 at the end represents the data title variation.
00 is specified when there is no particular indication of the version.
AOC_DATA_TITLE_VERSION Specifies the data title version.
AOC_DATA_TITLE_VARIATION Specifies the data title variation when one title makes use of multiple data titles. Specify a value in the range from 0 through 255. If omitted, 0 is specified. This is omitted in the demo.
AOC_BANNER_SPEC Specifies the BSF file to create the icon data.
AOC_CONTENT_INFOS Specifies the XML file to create the metadata content.
AOC_CONTENT_ICONS_DIR Specifies the directory where the icon image for the add-on content is located.
AOC_CONTENT_DIRS Specifies multiple directories where the add-on content files are stored. Content indices are assigned in order starting with 1.

Build Using Tools

The following tools are used to create data titles. For information about how to use tools, see the documentation for each tool. For examples of using tools, see SampleDemos/ec/DataTitleRaw.

ctr_makebanner
Creates icon data (.icn). Does not require a banner file (.bnr).
ctr_makeMetaDataContent
Creates metadata content (.cfa).
ctr_makerom
Creates add-on content (.cfa).
ctr_makecia
Creates a data title (.cia) to store both the metadata content and add-on content.

The following shows how to use ctr_makerom to create the CFA file. Options that are not shown here do not need to be specified. 

# Creation of add-on content (index 1).
ctr_makerom \
    -f data \
    -rsf $(CTRSDK_ROOT)/resources/specfiles/AddOnContent.rsf \
    -o objects/aoctitle_content_1.cfa \
    -DUNIQUE_ID=$(UNIQUE_ID)
    -DDATA_TITLE_INDEX=0
    -DPRODUCT_CODE=$(PRODUCT_CODE)
    -DTITLE=$(TITLE)
    -DROMFS_ROOT=../DataTitle/content01 \

With ctr_makecia, program arguments can be used to specify the CFA files to store in the data title, but when there is a large amount of content, you may not be able to include any program arguments. For this reason, ctr_makecia supports specifying CFA files with a content list file. The content list file is written in the following format. This example is excerpted from SampleDemos/ec/DataTitleRaw/cfalist.yaml.

-
  Index: 0
  Filename: objects/aoctitle_content_0.cfa

-
  Index: 1
  Filename: objects/aoctitle_content_1.cfa

-
  Index: 2
  Filename: objects/aoctitle_content_2.cfa

This format differs from the general YAML format. See the sample demo (SampleDemos/ec/DataTitleRaw/cfalist.yaml) when creating a content list file.

The following shows how to create a data title with ctr_makecia using a content list file. Options that are not shown here do not need to be specified.

ctr_makecia
    -o images/aoctitle.cia
    --datatitle-version 0
    -content-list cfalist.yaml

The RSF File to Use for Creating Add-On Content

To create add-on content, use the following RSF file and no other: CTR_SDK/resources/specfiles/AddOnContent.rsf.

Specify the following values for this AddOnContent.rsf file using the -Dname=value option of ctr_makerom or ctr_makeMetaDataContent. Specify all of these values when using ctr_makerom or ctr_makeMetaDataContent.

Variable Name Description
TITLE Specifies the application title in eight or fewer ASCII characters.
This must match the value of the TITLE specified in the game application's RSF (which uses the data title).
PRODUCT_CODE Specifies the product code.
UNIQUE_ID Specifies a unique ID.
DATA_TITLE_INDEX Specifies the data title variation. Normally, specify 0.
If you are using several data titles, you must set this to an appropriate value. For more information, see Creating and Using Multiple Data Titles.
ROMFS_ROOT Specify the root directory for where the data for the add-on content is located.

Handling a Large Volume of Add-On Content

When a single application has several hundred or several thousand pieces of add-on content, time is required to create the data title.

To shorten build times and reduce the time from revising add-on content to confirming the content, create data titles that store only the minimal add-on content that is required to verify the content of the metadata content, and then import this minimal add-on content using DevMenu during the development of add-on content.
