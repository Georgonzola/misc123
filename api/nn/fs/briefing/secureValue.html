Overview

On the CTR, downloaded applications are stored on the SD card. This makes it easy for users to back up their save data on a PC, but also makes it relatively easy to accidentally overwrite game save data at a later time. You can use the features provided in the SDK for preventing rollback of save data to avoid ruining the game experience by accidentally overwriting (rolling back) save data.

Save data for card applications is stored in a dedicated region in CTR memory. You do not need to use the save data rollback prevention feature for this save data because users cannot easily roll it back by accident. The SDK can automatically determine what kind of application is being used, and it can decide whether to use the rollback prevention feature because the source code and binary data for card applications and downloadable applications are essentially identical. As a result, if there is any chance that the application you are developing may be distributed as a downloadable application in the future and may use the save data rollback prevention feature (even if you only plan on releasing it as a card application for now), we strongly recommend that you develop the application using the save data rollback prevention API to add support for the feature ahead of time, and also fully debug your application in the download distribution format.

Usage

Secure Value Region

A 64-bit region known as a secure value region is created in system memory for each application for use with the save data rollback prevention feature. You can use the save data rollback prevention feature API to write values to and verify values from the secure regions. If you want to prevent save data rollback for a particular application, you can also use the API to associate a value in the secure region with a value in the application.

Saving Save Data

To prevent save data rollback for an application, create a 64-bit secure value when saving game data. There are no restrictions on the values that can be saved in the secure region, but we recommend the following methods to prevent duplicate saves.

  • Increment the value previously saved in the region.
  • Generate a random value.
  • Create a hash for the data to save.

After the secure value is created and written to the save data, specify the secure value and the archive name when calling the nn::fs::CommitSaveDataAndSetSecureValue function to commit the save data and overwrite the console's secure region. (If you are using the save data rollback prevention feature, do not use the nn::fs::CommitSaveData function.) If the power is turned off or the SD card is removed while this function is being executed, the secure values saved by both of these functions will be inconsistent. Make sure to display an appropriate message to users during this time to prevent this from happening. Do not allow the system to enter Sleep Mode or transition to the HOME Menu during this time.

For card application save data, calling the nn::fs::CommitSaveDataAndSetSecureValue function does not write a value to the secure region in system memory. Instead, the value is written to a region managed by the SDK on the CTR card. This value is used by a tool that can transfer card application save data so that it can be used with the download version of the application. When the save data is transferred, the value is copied to the secure value region in system memory.

Using the nn::fs::CommitSaveData and nn::fs::SetSaveDataSecureValue functions together to commit the save data and overwrite the secure region at the same time is no longer recommended. The rate at which inconsistencies occur due to interruptions during processing with this not-recommended approach is higher than when using the newly introduced nn::fs::CommitSaveDataAndSetSecureValue function.

To overwrite the secure area without committing the save data, call the nn::fs::SetSaveDataSecureValue function instead of the nn::fs::CommitSaveDataAndSetSecureValue function.

Loading Save Data

When loading save data, first retrieve the secure value from the save data. Pass this value to nn::fs::VerifySaveDataSecureValue to compare it with the secure value stored in system memory. This function returns a bool that indicates whether the values match. If this function returns false, the save data may have accidentally been rolled back. If so, make sure the application determines what to do next. For example, it could do one of the following.

  • Treat the save data as invalid and delete it.
  • Make part of the save data invalid.
  • Display a warning.

If you choose to have the application make some or all of the save data invalid, make sure to allow users to confirm whether to complete the operation. This situation may arise if the same application has been downloaded to more than one SD card and the user inserts the wrong SD card. However, the issue typically resolves if the user inserts the correct SD card and starts the application again.

Do not call the nn::fs::VerifySaveDataSecureValue function if the application save data is the factory default data, if it is corrupted, or if the secure value was not successfully written in the save data.

For card application save data, the nn::fs::VerifySaveDataSecureValue function always returns true for any secure value passed to it because the save data rollback prevention feature is disabled.

Loose Rollback Prevention

If you used a system to play applications that enabled the save data rollback prevention feature, and that system breaks and the values in the secure regions in system memory cannot be restored during the repair process, the secure values in the save data on the SD card do not match the secure values in system memory because they have not been written in the system. In this case, you can use the nn::fs::VerifySaveDataSecureValueLoosely function, which returns true for CTR systems on which the secure values have not yet been written, in place of the nn::fs::VerifySaveDataSecureValue function. Using the nn::fs::VerifySaveDataSecureValueLoosely function instead of the nn::fs::VerifySaveDataSecureValue function to verify secure values allows you to treat save data as if it had never been rolled back even on CTR systems that have been repaired.

In general, we recommend emphasizing usability and using Loose Rollback Prevention, but when using the nn::fs::VerifySaveDataSecureValueLoosely function, there is the problem of being able to freely roll back data on systems just after they have been repaired. If rollback of save data in general would be more problematic for your game application than the issue that occurs on the repaired system, use the nn::fs::VerifySaveDataSecureValue function.

Secure Value Mask

The nn::fs::SetSaveDataSecureValue function has an overload that also takes a 64-bit mask. When you specify this overload, only the bits of the secure value that are set in the mask are written to the secure value region. Similarly, for the nn::fs::VerifySaveDataSecureValue function that has an overload that takes masks, it only compares the bits of the secure value for which the bits of the mask were set and returns true if the bits that were set match. By using masks, you can emulate having multiple secure values by dividing up the 64 bits that comprise them.

Masks cannot be used with the nn::fs::VerifySaveDataSecureValueLoosely function. Accordingly, because masks cannot be set when using Loose Rollback Prevention, call the overload that does not take a mask specification when calling the nn::fs::SetSaveDataSecureValue function.

Whether to Specify an Archive Name

In the API for the save data rollback prevention feature, some overloads take archiveName as an argument and others do not.

You can use the overload that does not take archiveName as an argument when you are using the save data rollback prevention feature on save data for your application that is mounted with the nn::fs::MountSaveData function without a uniqueId parameter.

You must use the overload that takes archiveName as an argument when you use the save data rollback prevention feature on save data for another application that is mounted with the nn::fs::MountSaveData function with a uniqueId parameter. This overload can also be used on save data for your own application.

Debug Support

You can save or delete rollback prevention values by using SaveDataFiler. For more information, see SaveDataFiler.
