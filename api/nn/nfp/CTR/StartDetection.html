nn::nfp::CTR::StartDetection Function

Syntax

nn::Result StartDetection();

Parameters

None.

Return Values



Value Description
ResultSuccess Started detecting the tag.
ResultInvalidOperation The library is not in the correct state.
ResultSleep The system is in a sleep state.
ResultWifiOff The Wi-Fi switch is set to off.
Other Unexpected error.

Description

Starts detecting a tag.

Before you use this function, call SetActivateEvent and SetDeactivateEvent to receive the events for detected and lost tags. Tag-detection events can be delayed by as much as 0.5 seconds from when the tag was placed, and tag-lost events can be delayed by as much as 2 seconds after the tag was removed. The act of calling this API function also takes a few dozen milliseconds.

The electromagnetic waves that serve for tag detection are signal noise to SNAKE, so touch panel input is unstable until detection is stopped by StopDetection. If you use touch panel input under this signal noise condition, it might cause the application to malfunction, so SNAKE restricts the use of the touch panel while tags are being detected. To be specific, the status of nn::hid::CTR::TouchPanelStatus is always 0 (stylus up) for the time between the calls to StartDetection and StopDetection. Because CTR uses an accessory to detect tags, touch panel input does not become unstable, so it does not have the same kind of restriction imposed as SNAKE does. Note, however, that you might need to take special measures, such as designing a different UI for SNAKE where touch panel input is unavailable.

Power consumption and heat levels are highest at the moment an NFC tag is discovered and Activate is triggered in the NFP library. If StartDetection and StopDetection are called repeatedly while a tag is placed on the system, this Activate will be issued again and again over a short period of time, generating more heat than SNAKE is designed to tolerate and destabilizing the system. This overheating problem does not arise on the CTR, but the process consumes power the same way. For this reason, there are system limitations on both SNAKE and CTR for the frequency of calls to StartDetection. You can call StartDetection up to 10 times during a 10-second interval, but do not call the function any more frequently than that. This restriction is realized by blocking any call to StartDetection until 10 seconds have passed since the tenth previous call to the function. There is no blocking if 10 seconds have already passed, or if less than 10 calls have been made in the past.

On the CTR, if this function is called the moment you disconnect from the NFC reader and writer, ResultInvalidOperation could occur.
When that happens, execute the Connect function over again, after explicitly disconnecting by calling the Disconnect function.
