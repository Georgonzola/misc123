nn::rdt Namespace

Description

The namespace of the library for reliable local communication.

Overview: The RDT library is a class library that uses the features of the UDS library to implement reliable data communication (that is, communication without data loss or corruption). (Only Receiver knows whether all data was received. For more information, see the note later in this section.)

In addition, RDT was designed to send and receive relatively large amounts of data (more than a few dozen KBs). It does not perform very well in situations where small chunks of data (under 1 KB) are frequently being sent and received.

Sender class: Class that represents a device sending data. A Sender sends data to instances of the Receiver class (described below). The Sender class implements retransmission and other features that ensure that data that is sent is always received, even if errors like packet loss occur during communication.

Receiver class: Class that represents a device receiving data. A Receiver class receives data from an instance of the aforementioned Sender class. The receiver sends appropriate response confirmations to the sender and the arrived data is reconstructed in the correct order, so the byte string obtained by the application is guaranteed to be preserved in the same order as the byte string that was sent.

UDS setup: Before an application can use RDT, it must first set up UDS. In other words, do not initialize the RDT library until a network has been created, the local host has connected to that network, and the transmission of data via UDS has become possible. Setting up UDS communication is the responsibility of the application. The same is true for dismantling the network that was created.

Features: The RDT class library involves a unidirectional flow of raw data from the Sender to the Receiver. If bidirectional communication is required, you must create another Sender/Receiver pair.

Non-blocking operation: Function calls in the RDT class library do not block. Calls to member functions such as Sender::Open, Receiver::Wait, Sender::Send, and Receiver::Receive return immediately. The actual sending and receiving is handled by the Process function that has been provided for both classes.

Process function: This function must be called at least at intervals of each game frame (that is, less than 50 milliseconds) until the application calls the Sender::Finalize and Receiver::Finalize functions. If Process is not called periodically, no actual communication or state transitions occur. (Although a few functions, including Cancel, are exceptions to that rule.) You can sometimes improve performance of sending and receiving data by calling the Process function several times in a row.

Memory allocation: The RDT library does not allocate memory dynamically. Any memory required by the library must be provided to the library by the application when Initialize is called. Note: When providing this memory, pay attention to its alignment.

Note: Only Receiver knows whether all data was received. Only when Receiver is in a state of having completed receiving data (RECEIVER_STATE_WAITING_FINISHED or RECEIVER_STATE_FINISHED) can all data be considered received. There is no guarantee that Receiver has received data even if Sender enters the send complete status (SENDER_STATE_CLOSING or SENDER_STATE_CLOSED). As a consequence, to use the RDT library to make certain that data sending and receiving takes place requires a session that controls data sends and receives in addition to sending data with one Sender and Receiver set. For more information, see the sample demos.

The RDT library implicitly consumes two nn::uds::EndpointDescriptor objects per each Sender or Receiver instance.

Classes

nn::rdt::Receiver Class representing a device receiving data. The following description shows how to use the Receiver class. Detailed instructions on calling Process have been left out of the following description. In actual practice, Process must be called periodically at a frequency of about once per game frame or less.
nn::rdt::Sender Represents a device that sends data. The following description shows how to use the Sender class. Detailed instructions on calling Process have been left out of the following description. In actual practice, Process must be called periodically at a frequency of about once per game frame or less.
nn::rdt::ResultAlreadyInitialized Indicates that a function exited without doing anything because initialization had already finished. (This is not an error.)
nn::rdt::ResultDoNothing Indicates that a function exited without doing anything. (This is not an error.)
nn::rdt::ResultNotInitialized Indicates an uninitialized instance.
nn::rdt::ResultSendBufferIsNotAvailable Indicates that the Sender::Send function attempted to write data to the send buffer, but the send buffer did not have enough free space.
nn::rdt::ResultInvalidSize Indicates that an invalid size was specified as an argument.
nn::rdt::ResultMisalignedAddress Indicates that a memory address with an invalid alignment was passed as an argument.
nn::rdt::ResultNullPointer Indicates an error caused by specifying a null pointer as an argument.
nn::rdt::ResultResetReceived Indicates that a reset signal was received from a peer, and there was a state transition to CLOSED. The reset signal is sent in response to mistaken connection requests and when Sender::Cancel or Receiver::Cancel is executed. When this reset signal is received, the instance changes state to CLOSED. Depending on the situation, you may want to handle this by trying to reconnect again or by disconnecting altogether.
nn::rdt::ResultResendQueueIsFull Indicates that the internal resend queue for the RDT library was full.
nn::rdt::ResultUntimelyFunctionCall Indicates that a function was called at the wrong time. Call some functions, such as Sender::Open and Receiver::Wait, only at particular times or instance states. For example, do not call Sender::Open when the instance is already in the SENDER_STATE_OPENED state. This result code is returned in such situations.
nn::rdt::ResultNoData A result code that is used only within the library. Indicates that no segments have been received.
nn::rdt::ResultInvalidValue A result code that indicates that an argument value was invalid.
