nn::uds::CTR::NetworkDescription::GetTemporaryId Member Function

Syntax

u32 GetTemporaryId() const;

Parameters

None.

Return Values

Returns the TemporaryId.

Description

Gets the temporary ID. This ID can be used to identify the network, but instead we recommend that you use the BSSID or the host's NodeInformation. When a device builds a network and operates as the host, usually a random number is generated and used as the TemporaryID for that network so that the network can be distinguished from a network built by the same device and on which local communications have taken place, but on which communications have been broken and resumed.

However, if a TemporaryID is generated every single time a network is built, consider what happens when you implement an algorithm such as the following to build networks without the end user knowing whether they are host or client.

1: Become a host (build a network).
2: If you cannot perform client connection after a specified period of time, destroy the network.
3: Search for nearby networks. If a network for the same application exists, connect to that network as a client. If you cannot find a peer to connect with, repeat these steps.

With this kind of algorithm, devices try to connect based on their search results, but because a TemporaryID is generated every single time a network is built, peers frequently cannot connect because the host has rebuilt its network. The upshot is that the implementer is not able to realize the expected operations.

UDS communication provides for the following exception.

Do not update the TemporaryID when rebuilding the network if no clients are connected and communication between devices has not been established.
(However, if it takes more than 60 seconds to rebuild the network, assume the network being built has no relation to the previous network and go ahead and update the TemporaryID.)


Contact Nintendo if there is a problem implementing this specification in your application.
