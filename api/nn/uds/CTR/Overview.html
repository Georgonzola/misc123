nn::uds::CTR Namespace

Description

The namespace of the CTR library for performing local communication. (It can also be written without the CTR portion.)

This library is used to run UDS communication, Nintendo's unique local wireless communication. For more information about UDS communication, see the programming manuals.

Notes

Function Blocking

With the exception of the GetLinkLevel function, all of the UDS library functions can be blocked by several milliseconds, depending on the situation. (As stated in the API Reference, if the block time is greater than 100 ms it is the reference value.)

Note that this can cause problems such as unstable frame rates when these functions are called within the main thread.

Sleep Mode During UDS Communication

Sleep Mode Is Prohibited During Initialize

Do not enter Sleep Mode while the Initialize function is running. The process of transitioning the communication system to UDS mode may conflict with background communication processing when the system enters Sleep Mode, resulting in incorrect operations.

Sleep Mode Transitions During Communication

Because the system cannot maintain UDS communication during Sleep Mode, applications must shut down the UDS library before allowing the system to enter Sleep Mode.

If the system enters Sleep Mode without shutting down the UDS library, the UDS library forces all communication in progress to end immediately before the entire system enters Sleep Mode, resulting in an internal state of STATE_ERROR. Once the library has fallen into this state, all functions except for a few (such as Finalize and GetConnectionStatus) return an error.

Guaranteeing Packet Arrival

The UDS library itself does not guarantee that packets have actually been sent to the destination. A packet loss of approximately 1-2% is normal and in some cases may be 10%. Do not use code that assumes the packets you send will always reach the destination.

Packet Order

As long as packets are sent to the same node with the same options specified, they are guaranteed to be sent in the same order. When the packets are addressed to BROADCAST_NODE_ID or certain nodes, the order in which they are sent and received may be swapped. Similarly, the order in which they are sent and received may also be swapped when you switch between using the FORCE_DIRECT_BC and FORCE_UNICAST options.

Specification for TemporaryID With UDS Communication

When a device builds a network and operates as the master, usually a random number is generated and used as the TemporaryID for that network so that the network can be distinguished from a network built by the same device and on which local communications have taken place, but on which communications have been broken and resumed.

However, if a TemporaryID is generated every single time a network is built, consider what happens when you implement an algorithm like the following one to build networks without the end user keeping track of whether they are the host or the client.

1: Become a host (build a network).
2: If you cannot perform client connection after a specified period of time, destroy the network.
3: Search for nearby networks. If a network for the same application exists, connect to that network as a client.
If you cannot find a peer to connect with, repeat these steps.

With this kind of algorithm, devices try to connect based on their search results,
but because a TemporaryID is generated every single time a network is built,
peers frequently cannot connect because the host has rebuilt its network.

The upshot is that the developer's implementation cannot achieve the expected operations.

UDS communication provides for the following exception.

Do not update the TemporaryID when rebuilding the network if no clients are connected and communication between devices has not been established.
(However, if it has been a long time since the network was rebuilt, assume that the network being built has no relation to the previous network and go ahead and update the TemporaryID.)



Contact Nintendo if there is a problem implementing this specification in your application.

Communication Between CTR and SNAKE Systems

The SNAKE and CTR processors run at different speeds, and this can lead to communication problems.
Do not implement communication process that greatly depend on processing speed.
For example, use a timer to introduce an appropriate delay after sending a packet to avoid excessive sending of packets.
Alternatively, implement a process whereby the sender checks on the receiving status of the receiver as needed.
When testing operations, use the CTR and SNAKE systems in numerous combinations to confirm that no problems will arise.

Classes

Handling
nn::uds::CTR::NetworkDescription Class indicating network information that can be obtained using nn::uds::CTR::NetworkDescriptionReader.
nn::uds::CTR::NetworkDescriptionReader Class for parsing all types of network information that can be obtained using nn::uds::CTR::ScanResultReader.
nn::uds::CTR::ScanResultReader Class for parsing the scan results obtained by nn::uds::CTR::Scan.
Result
nn::uds::CTR::ResultNotFoundNetwork Indicates that the network to which a connection was attempted using ConnectNetwork was not found.
nn::uds::CTR::ResultAlreadyNetworkIsFull Indicates that the network to which a connection was attempted using ConnectNetwork is full. (The number of devices connected to that network has already reached the maximum.)
nn::uds::CTR::ResultDeniedFromMaster Indicates that the attempt to connect to the network using ConnectNetwork was denied by the host. If the host application is not intentionally prohibiting connection, this error may also occur when the connection cannot be made because of poor signal quality.
nn::uds::CTR::ResultConnectionTimeout The connection to the network using the ConnectNetwork function timed out.
nn::uds::CTR::ResultInvalidState Indicates that the function cannot run in the current library state. This result can also occur if the connection is lost during communication.
nn::uds::CTR::ResultInvalidNode Indicates that the node with the specified node ID does not exist on the network. This result can also occur when the node attempts to send a packet to itself, or when the connection target disconnects from the network.
nn::uds::CTR::ResultOutOfResource Indicates that the library is out of resources. This result can occur if the total size of the receive buffer specified by the Attach function exceeds that of the buffer specified when the Initialize function was called.
nn::uds::CTR::ResultBufferIsFull This mostly occurs if SendTo is called with high frequency, and send operations get backed up. The data being sent is lost if an error occurs.
nn::uds::CTR::ResultWirelessOff Indicates that the system entered wireless-disabled mode after the UDS library was initialized.
nn::uds::CTR::ResultNotInitialized Indicates that the library has not been initialized. This error can be resolved by calling the Initialize function.
nn::uds::CTR::ResultAlreadyOccupiedWirelessDevice UDS communication cannot be used at the moment because other communication is already in progress.
nn::uds::CTR::ResultOutOfRange Indicates that one or more of the arguments were set to values outside their valid ranges.
nn::uds::CTR::ResultMisalignedAddress Indicates that the address alignment is invalid.
nn::uds::CTR::ResultMisalignedSize Indicates that the size alignment is invalid.
nn::uds::CTR::ResultTooLarge Indicates that one or more of the arguments had data that exceeded the maximum size. This error occurs for functions that send data.
nn::uds::CTR::ResultNotAuthorized Indicates that one or more of the arguments contain a value that is permitted only in special cases.
nn::uds::CTR::ResultMiscellaneousSystemError A temporary failure resulting from something on the system side. Normally, if you retry with the same arguments, the function succeeds.
nn::uds::CTR::ResultNotImplemented The function is defined but not implemented.
nn::uds::CTR::ResultMalformedData Indicates that the data is not correct, and that the received data may have been tampered with.
nn::uds::CTR::ResultInvalidData Indicates that the received data is invalid.

Structures

nn::uds::CTR::ConnectionStatus Stores the UDS connection status.
nn::uds::CTR::ScrambledLocalFriendCode Stores information that can be converted to a local friend code using the friends library. To simplify the use of friend features in local communications and at the same time protect user privacy and prevent individual tracking, you can query the friends library and get the local friend code as represented by a data array of nn::uds::CTR::SCRAMBLED_LOCAL_FRINED_CODE_SIZE bytes. This way you do not use the local friend code from the UDS library.
nn::uds::CTR::NodeInformation Stores user information about nodes connected to the network.
nn::uds::CTR::EndpointDescriptor This is an endpoint descriptor. Corresponds to a socket descriptor.

Enumerated Types

State Enumerates UDS states.
DisconnectReason Enumerates the reasons for disconnection.
ConnectType Enumerated type that represents modes used during connection to the network.
PowerSaveMode Enumerates the network's power-saving modes.
LinkLevel Enumerated type indicating the communication quality (link level).

Constants

Alignment
S BUFFER_ALIGNMENT const s32 Specifies the number of bytes that the address of the received buffer specified in Initialize must be aligned to. It is defined as 4096 bytes.
S BUFFER_UNITSIZE const size_t Specifies the number of bytes that the size of the receive buffer specified in Initialize must be a multiple of. It is defined as 4096 bytes.
S SEND_DATA_ALIGNMENT const s32 Specifies the number of bytes that the starting address for where to store outgoing data specified in SendTo must be aligned to. It is defined as 4 bytes.
S SEND_DATA_UNITSIZE const s32 Specifies the number of bytes that the size of the region for where to store outgoing data specified in SendTo must be a multiple of. It is defined as 4 bytes.
S RECEIVE_DATA_ALIGNMENT const s32 Specifies the number of bytes that the starting address for storing incoming data specified in Receive must be aligned to. It is defined as 4 bytes.
S RECEIVE_DATA_UNITSIZE const size_t Specifies the number of bytes that the size of the region for where to store incoming data specified in Receive must be a multiple of. It is defined as 4 bytes.
Transmission Data Sizes
NET_DESC_APPDATA_SIZE_MAX const u8 Maximum size of the application data that can be set for the beacon.
UDS_PACKET_PAYLOAD_MAX_SIZE const u16 Maximum size of the data that can be sent in a single call to nn::uds::CTR::SendTo.
ATTACH_BUFFER_SIZE_MIN const size_t Minimum size of the receive buffer specified in a call to the Attach function. This value is the sum of the UDS_PACKET_PAYLOAD_MAX_SIZE and the system management region.
ATTACH_BUFFER_SIZE_DEFAULT const size_t Default size of the receive buffer specified in a call to the Attach function.
String Length of the Passphrase
UDS_PASSPHRASE_LENGTH_MIN const size_t Minimum size of the passphrase for generating the encrypted key used in communication.
UDS_PASSPHRASE_LENGTH_MAX const size_t Maximum size of the passphrase for generating the encryption key used in communication.
Send/Receive Options
NO_WAIT const bit8 If this constant is specified as an argument to the SendTo function, wireless transmission occurs immediately without buffering within the UDS library. If this constant is specified as an argument to either the Receive or ReceiveFrom functions, the wireless transmission ends immediately, even if no data has been received.
FORCE_DIRECT_BC const bit8 When SendTo has been specified, sends using direct broadcast without asking for a recipient. Unicast communication among clients takes place without the use of a host. As a result, you can have low latency communication, but packets may not arrive.
Other
BROADCAST_NODE_ID const u16 Node ID representing all node addresses of the network. Data sent to BROADCAST_NODE_ID can be retrieved by Spectators.
NODE_MAX const u8 Maximum value for the maximum number of network connections. In the future, we plan to support communication for 16 devices, but as of CTR-SDK 0.14 we cannot confirm sufficient operations with that many devices, so communications with 13 or more devices exceeds the guaranteed range of operations.
ENDPOINT_MAX const u8 The maximum number of endpoints that can be created. This value is subject to change in the future.
S SCRAMBLED_LOCAL_FRINED_CODE_SIZE const size_t The byte length of ScrambledLocalFriendCode.

Functions

Initialization and Shutdown
Initialize Initializes the UDS library. Initialization fails if any other communication features are already in use.
Finalize Finalizes the UDS library. After executing, other communication features become usable.
Networks (Creating, Destroying, Connecting, and Disconnecting)
CreateLocalCommunicationId Creates a local communication ID from a unique ID. This function uses a 20-bit unique ID assigned by Nintendo to generate a 32-bit value (a local communication ID) to use for UDS communication.
If Nintendo has not assigned a unique ID to you, use one of the unique IDs for game prototypes (in the range from 0xFF000 through 0xFF3FF).
However, you must get a unique ID from Nintendo for your retail product.

This function is thread-safe.
CreateNetwork Creates a new network.
Scan Scans for nearby networks.
ConnectNetwork Connects to an existing network.
EjectClient Kicks the specified node off the network. Only the host can call this function.

This function is thread-safe.
DisallowToConnect Prevents client nodes from connecting to the network. This function does not affect clients that are currently connected.
AllowToConnect Allows the client to connect to the network.
EjectSpectator Kicks all connected spectators off the network.
AllowToSpectate Allows the spectator to connect to the network.
DestroyNetwork Destroys the network.
DisconnectNetwork Disconnects the local node from the network.
Functions for Sending and Receiving Data
CreateEndpoint Creates a network endpoint and returns the corresponding descriptor.
SetMaxSendDelay Specifies the maximum send delay time.
SendTo Sends data to the specified port on the specified partner's system.
Attach Enables the local device to receive packets from the endpoint. Assigns the specified port and node ID to the endpoint. The system creates a receive buffer and prepares to receive packets at the port specified below.

This function is thread-safe.
Receive ‏Receives data. (No functionality is provided for getting the address of the sender.)
ReceiveFrom ‏Receives data.
DestroyEndpoint Destroys an endpoint.
Other
GetConnectionStatus Gets the current connection status.
GetLinkLevel Gets the current link level.
GetChannel Gets the current channel.

This function is thread-safe.
GetNodeInformation Gets information about the specified node.

This function is thread-safe.
SetApplicationDataToBeacon Sets application data in the beacon.
GetApplicationDataFromBeacon Gets the data set in the beacon.
